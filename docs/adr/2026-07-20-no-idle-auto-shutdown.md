# ADR: 不为 scholia serve/start 增加空闲自动退出

**日期**：2026-07-20
**状态**：已采纳

## 背景

`scholia start` 会把 `serve` 拉到后台常驻运行（详见 [../../README.md](../../README.md) 的 `scholia start` 一节）。讨论中提出：既然服务可能被启动后长期挂着不用，是否应该加一个空闲自动退出机制来节省资源。

## 决策

不加。`scholia serve`/`start` 保持常驻运行，直到用户显式执行 `scholia stop`。

## 理由

- **空闲时的资源占用可忽略**：Koa/Node 进程空闲时只占用几十 MB 内存、CPU 接近 0，本地个人工具场景下不构成实际资源压力；真正消耗资源的是视频播放/大文件传输，这种情况下服务并不空闲。
- **服务端无法可靠判断"空闲"**：scholia 没有浏览器侧的心跳/在线信号，只能靠请求间隔粗略推测。用户打开一篇文章或一段视频后长时间不操作页面（阅读、看视频）在服务端看来和真正离开是一样的，自动退出容易在用户还在使用时把会话误杀。要做对，需要额外引入前端心跳上报机制，复杂度和收益不成比例。
- **已有更明确的生命周期控制**：`scholia start` 配合 `scholia stop` 已经能让用户按需精确控制何时启停，足够覆盖"忘记停止导致端口一直占着"这个担忧，不需要再叠加一套自动化机制。

## 后果

- 用户需要自己记得用 `scholia stop` 结束会话；长期不关闭也不会造成明显资源问题。
- 如果未来出现新的资源占用场景（例如常驻大量后台任务），需要重新评估这个决策。
